# SlashTablet — концепт для дизайна

> **Статус:** концепт / design brief  
> **Продукт:** 1Dent CRM  
> **Аудитория документа:** дизайнер, продукт  
> **Важно:** это описание намерений. Код и существующие экраны CRM **не меняются** на этом этапе.

---

## 1. Что такое SlashTablet

**SlashTablet** — отдельная страница (отдельный «режим» продукта) для врачей, оптимизированная под **планшет в кабинете**. Врач открывает её с планшета в приёмной, а не через обычный мобильный кабинет врача в телефоне.

**Главная идея:** планшет в кабинете = рабочее место врача у кресла. На нём врач ведёт приём: выбирает пациента, показывает зубы, объясняет план лечения, демонстрирует видео — всё в одном фокусированном интерфейсе **без лишних разделов CRM**.

| Сейчас в CRM | SlashTablet |
|---|---|
| Врач заходит с телефона в «Кабинет врача» | Врач заходит с **планшета в кабинете** |
| Пациенты — узкая боковая панель (`max-w-md`) | Пациент — **полноэкранная рабочая область** (~960px и шире) |
| Много разделов: расписание, аналитика, чат, зарплата | **Только пациенты и клиническая работа** |
| Обычный логин | Вход через **QR-код кабинета** |

---

## 2. Проблема, которую решаем

Врач в кресле должен:

1. Быстро открыть нужного пациента (сегодняшний приём).
2. Показать пациенту понятную картину: какие зубы в каком состоянии.
3. Объяснить и «продать» план лечения — наглядно, без мелкого текста на телефоне.
4. При необходимости включить **обучающее / демонстрационное видео** (процедура, гигиена, имплантация и т.д.).
5. Работать **касанием**, стоя или сидя у кресла, не отвлекаясь на финансы, чат и админку.

Телефонный интерфейс 1Dent для врача хорош для личного использования вне кресла. **SlashTablet** — для **рабочего планшета в кабинете**, ориентированного на совместную работу врача и пациента.

## 3. Как врач попадает на страницу (QR-сценарий)

### 3.1 Два типа QR в системе

| QR | Где | Кто сканирует | Результат |
|---|---|---|---|
| **QR кабинета** (статичный) | Наклейка / табличка на стене или у монитора в кабинете | Врач (первый раз за смену или после выхода) | Открывается **SlashTablet**, привязанный к этому кабинету/филиалу |
| **QR сессии** (опционально, v2) | На экране планшета | Пациент | Только просмотр презентации плана (без редактирования) |

На первом этапе дизайнируем **QR кабинета + вход врача**.

### 3.2 Пользовательский поток (врач)

```
┌─────────────────────────────────────────────────────────────────┐
│  1. В кабинете на стене — QR-код «Кабинет 3 · 1Dent»            │
└───────────────────────────────┬─────────────────────────────────┘
                                │ сканирование
                                ▼
┌─────────────────────────────────────────────────────────────────┐
│  2. Планшет открывает SlashTablet                                │
│     URL вида: /tablet?cabinet=...  (точный путь — на этапе dev) │
│     Контекст: филиал + кабинет (если заведён в системе)          │
└───────────────────────────────┬─────────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────────┐
│  3. Если врач не авторизован — быстрый вход                      │
│     · PIN / Face ID на планшете (локальная сессия)               │
│     · или выбор врача из списка клиники + PIN                    │
│     · или уже залогинен с прошлой смены                          │
└───────────────────────────────┬─────────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────────┐
│  4. Главный экран SlashTablet — список пациентов врача           │
│     (сегодня / текущая смена / поиск)                            │
└───────────────────────────────┬─────────────────────────────────┘
                                │ тап по пациенту
                                ▼
┌─────────────────────────────────────────────────────────────────┐
│  5. Карточка пациента — зубная карта, план, видео, презентация   │
└─────────────────────────────────────────────────────────────────┘
```

### 3.3 Что должен «знать» QR

Минимальный набор данных в QR (для дизайна — показать в макете наклейки):

- Клиника / филиал (название, адрес — для подтверждения «вы в правильном кабинете»).
- Идентификатор кабинета (Кабинет 1, 2, 3… или имя врача, если кабинет персональный).
- Ссылка на SlashTablet (не на общий логин CRM).

**Дизайн-задача:** макет **физической наклейки QR** (A6 / 10×10 см): логотип 1Dent, название кабинета, короткая инструкция «Отсканируйте планшетом для входа в кабинет».

### 3.4 Сканер QR на планшете

На **первом экране SlashTablet** (или в настройках кабинета) — **встроенный сканер QR** (камера планшета), чтобы врач мог отсканировать код со стены, если открыл браузер вручную, а не по прямой ссылке.

Экраны для дизайна:

1. **Приветствие / пустое состояние** — «Отсканируйте QR в кабинете» + кнопка «Открыть камеру».
2. **Сканер** — рамка, подсказка, обработка ошибки (неверный QR, QR другой клиники).
3. **Подтверждение кабинета** — «Вы вошли в Кабинет 3, ул. Абая 12» + «Продолжить».

## 4. Информационная архитектура SlashTablet

### 4.1 Что ЕСТЬ на странице

| Раздел | Описание |
|---|---|
| **Пациенты** | Список / сетка пациентов врача на сегодня (+ поиск по имени/телефону) |
| **Карточка пациента** | Полноэкранная работа с одним пациентом |
| **Зубная карта (FDI)** | Интерактивная схема челюстей, состояния зубов, выбор зуба |
| **План лечения** | Этапы, позиции, цены, скидки — визуально для обсуждения с пациентом |
| **Видео** | Библиотека роликов по процедурам + воспроизведение на весь экран |
| **Режим «Показать пациенту»** | Упрощённый UI без служебных кнопок, крупная типографика, поворот к пациенту |

### 4.2 Чего НЕТ на странице (намеренно)

- Финансы, касса, Kaspi, долги (кроме итоговой суммы в плане, если нужно для согласования).
- WhatsApp-чат, рассылки.
- Аналитика врача, зарплата, KPI.
- Админка клиники, склад, payroll.
- Kanban воронки «новая заявка» (можно упростить до «мои пациенты сегодня»).

SlashTablet — **клинический инструмент у кресла**, не замена полного CRM.

### 4.3 Навигация (предложение)

```
┌──────────────────────────────────────────────────────────────┐
│  [Лого 1Dent]   Кабинет 3 · Др. Асанова    [Врач ▾] [⚙]   │  ← верхняя полоса
├──────────────────────────────────────────────────────────────┤
│                                                              │
│                    ОСНОВНОЙ КОНТЕНТ                          │
│                                                              │
├──────────────────────────────────────────────────────────────┤
│   Пациенты   │   (внутри карточки: Карта · План · Видео)    │  ← нижняя или боковая
└──────────────────────────────────────────────────────────────┘
```

**Вариант A (рекомендуемый):** список пациентов → полноэкранная карточка с **табами**: «Карта зубов» · «План лечения» · «Видео» · «Инфо».

**Вариант B:** split-view на широком планшете — слева список пациентов (узкая колонка), справа карта + план.

Дизайнеру: проработать **оба варианта** для landscape 1024×768 и 1280×800.

## 6. Режим «Показать пациенту» (ключевая фича UX)

Отдельная кнопка в карточке пациента: **«Повернуть к пациенту»** / **«Режим презентации»**.

| Врач видит | Пациент видит (тот же экран, другой слой) |
|---|---|
| Все инструменты редактирования | Только карта, план, видео |
| Мелкие служебные подписи | Крупные заголовки, простые слова |
| Цены в деталях | Итоговые пакеты «от … ₸» |
| FDI-номера (опционально) | «Верхний правый большой зуб» или схема без номеров |

**Поведение:** один тап переключает режим; экран можно физически повернуть к пациенту; жест «свайп вниз» или PIN врача возвращает в рабочий режим.

Дизайнеру: **два визуальных темы** одного экрана — «врач» и «пациент».

## 7. Устройства и вёрстка

| Параметр | Значение |
|---|---|
| Основной форм-фактор | Планшет **landscape** |
| Минимальная ширина | 768px (iPad mini) |
| Целевая ширина контента | **960–1200px** (см. WideView mockup) |
| Ориентация | Landscape приоритет; portrait — fallback с предупреждением «Поверните планшет» |
| Touch | Минимум **44×44px** для кнопок и зубов на карте |
| Safe area | Учесть `safe-area-inset` как в мобильном `app-layout` |

**Не копировать** узкую боковую панель пациента из текущего CRM (`patient-detail-panel` — `max-w-md`). SlashTablet — **широкий рабочий стол**.

---

## 8. Связь с существующим продуктом 1Dent

SlashTablet — **новая оболочка (shell)** поверх тех же данных, что уже есть в CRM:

| Функция CRM | Использование в SlashTablet |
|---|---|
| Пациенты, процедуры на день | Список на главном экране |
| FDI зубная карта (`FdiChart`) | Центральный элемент карточки |
| Планы лечения (`TreatmentStagesBoard`) | Вкладка «План» |
| Голосовой ввод диагноза | Опционально, v2 |
| Филиалы клиники | Контекст из QR |
| Роль `doctor` | Доступ только к своим пациентам (или кабинету) |

Дизайн должен **визуально совпадать** с 1Dent (`DESIGN_SYSTEM.md`): кремовый фон, синий акцент, те же радиусы и типографика — но **плотность и размеры** под планшет и дистанцию до пациента.

## 9. Роли и права (для контекста макетов)

| Роль | SlashTablet |
|---|---|
| **Врач** | Полный доступ: пациенты, карта, план, видео, режим презентации |
| **Ассистент** (v2) | Тот же кабинет, ограниченное редактирование |
| **Пациент** | Только просмотр в режиме презентации (без отдельного логина) |
| **Админ клиники** | Настройка QR кабинетов в основном CRM (не в SlashTablet) |

На первом этапе дизайна — **только врач**.

## 11. User stories (для сценариев в Figma)

1. **Как врач**, я сканирую QR в кабинете и за 10 секунд вижу своих пациентов на сегодня.  
2. **Как врач**, я открываю карточку пациента и показываю на большой схеме, какие зубы требуют лечения.  
3. **Как врач**, я собираю план лечения по этапам и показываю пациенту итоговую сумму без переключения в другие приложения.  
4. **Как врач**, я включаю видео про имплантацию, чтобы пациент понял процедуру.  
5. **Как врач**, я нажимаю «Показать пациенту» и поворачиваю планшет — интерфейс становится простым и читаемым с расстояния 1 метра.  
6. **Как врач**, после приёма я возвращаюсь к списку и открываю следующего пациента без повторного сканирования QR.

## 13. Открытые вопросы для продуктa / дизайна

1. **Кабинет** — отдельная сущность в админке (Кабинет 1, 2, 3) или QR только на филиал?  
2. **Общий планшет** на несколько врачей в смену vs персональный кабинет?  
3. **Видео** — встроенная библиотека 1Dent, YouTube, или загрузка клиникой?  
4. **Подписание плана** — в SlashTablet или остаётся в основном CRM / договорах?  
5. **Название в UI** — «SlashTablet», «Кабинет», «1Dent Tablet» — что показываем пользователю?

## 14. Краткое резюме для дизайнера

**SlashTablet** = планшет у кресла + QR на стене + только пациенты + большая зубная карта + наглядный план лечения + видео + режим презентации для пациента.

Не дизайним «ещё один дашборд врача». Дизайним **спокойный, широкий, touch-first инструмент консультации**, который врач открывает одним сканированием и не закрывает до конца смены.
