# **MVP (первая версия продукта)**

**Фиксация базового функционала системы**

MVP-версия системы предназначена для запуска минимально работоспособного продукта, который позволит клинике управлять пациентами, коммуникацией и базовыми медицинскими процессами в единой системе.

Основная цель MVP — автоматизировать поток пациентов, обеспечить контроль коммуникаций и предоставить врачам удобный инструмент работы, не перегружая систему сложными функциями полной версии.

В рамках MVP реализуется основной операционный контур клиники:

- прием заявок и пациентов
- управление процессом лечения
- коммуникация через WhatsApp
- базовая аналитика и контроль владельца
- защита базы пациентов

Все функции, не указанные ниже, относятся к Full версии системы и не входят в MVP.

# **1. Канбан-доска и мини CRM**

**(основа управления пациентами и заявками)**

В системе реализуется канбан-доска для управления потоком пациентов, которая выступает как простая CRM и заменяет сторонние инструменты (например Trello или аналогичные сервисы).

Канбан позволяет визуально управлять стадиями работы с пациентом — от первичной заявки до завершения лечения.

**Основной функционал канбан-доски**

- Отображение пациентов и заявок в виде карточек на канбан-доске
- Перетаскивание карточек между статусами
- Настраиваемые этапы воронки

Пример статусов:

- Новая заявка
- Первичная консультация
- Диагностика
- Назначено лечение
- Лечение в процессе
- Послеоперационный контроль
- Завершено

Каждая карточка пациента содержит:

- ФИО пациента
- номер телефона
- источник заявки
- лечащий врач
- текущий статус
- дата последнего взаимодействия

**Карточка пациента**

При открытии карточки пациента доступна полная информация:

Основные данные:

- ФИО пациента
- телефон
- возраст
- комментарии администратора
- лечащий врач
- статус лечения

История взаимодействий:

- сообщения WhatsApp
- заметки врача
- изменения статусов
- проведенные процедуры

История сохраняется и доступна для просмотра владельцу и администраторам.

**Роли пользователей**

В MVP реализуется базовая ролевая модель доступа.

Поддерживаемые роли:

Владелец

- полный доступ ко всей системе
- просмотр аналитики
- доступ к базе пациентов
- управление пользователями

Администратор

- управление заявками
- запись пациентов
- коммуникация с пациентами
- изменение статусов

Врач

- доступ только к своим пациентам
- работа с карточкой лечения
- заполнение медицинских данных

Бухгалтер

- просмотр финансовых операций
- учет процедур

Склад

- учет препаратов и материалов

# **2. Интеграция с WhatsApp и чат-бот**

Одним из ключевых каналов коммуникации с пациентами является WhatsApp, поэтому в MVP реализуется глубокая интеграция с WhatsApp Business API.

Система позволяет вести всю коммуникацию с пациентами прямо внутри CRM, без необходимости использовать отдельные мессенджеры.

**Встроенный чат**

В CRM реализуется интерфейс чата, похожий на WhatsApp.

Возможности:

- переписка с пациентом
- отправка сообщений
- получение сообщений
- отображение истории переписки
- прикрепление сообщений к карточке пациента

Администраторы и врачи могут видеть переписку пациента.

**Автоматический чат-бот**

Для автоматизации работы с пациентами реализуется базовый чат-бот WhatsApp.

Бот выполняет следующие задачи:

- автоматическое приветствие пациента
- первичный сбор информации
- уточнение типа проблемы
- передача диалога администратору
- сопровождение пациента после лечения

**Послеоперационный мониторинг**

Система автоматически отправляет пациенту сообщения через определенное время после процедуры.

Примеры:

- через 24 часа
- через 3 дня
- через 7 дней

Пациент отвечает на сообщения, и система анализирует ответ.

**Red Alert (система тревожных сигналов)**

Если пациент в ответе указывает опасные симптомы, система автоматически фиксирует тревожный статус.

Примеры триггеров:

- сильная боль
- кровотечение
- воспаление
- высокая температура

В этом случае:

- карточка пациента получает статус Red Alert
- врачу и администратору приходит уведомление
- пациент поднимается в приоритет обработки

# **3. Дашборды и прозрачность работы**

Для контроля работы клиники система включает разные дашборды для разных ролей.

Каждый пользователь видит только ту информацию, которая относится к его зоне ответственности.

**Dashboard владельца**

Панель владельца показывает общее состояние клиники.

Основные метрики:

- количество новых пациентов
- количество текущих лечений
- завершенные процедуры
- загрузка врачей
- Red Alert случаи

Цель панели — дать владельцу быстрое понимание ситуации в клинике.

**Dashboard врача**

Панель врача отображает:

- список его пациентов
- назначенные процедуры
- текущие лечения
- задачи на сегодня
- уведомления

Также отображается базовая статистика врача:

- количество пациентов
- выполненные процедуры
- текущие случаи лечения

**Dashboard администратора**

Панель администратора включает:

- поток новых заявок
- записи пациентов
- управление расписанием
- уведомления о Red Alert

Администратор отвечает за операционное управление потоком пациентов.

# **4. Система безопасности и защита базы пациентов**

**(анти-угонная модель)**

Критически важным требованием системы является защита базы пациентов клиники от утечки и “угона” врачами.

В MVP реализуется базовая анти-угонная архитектура.

**Ограничение доступа к контактам**

Врач не видит полный номер телефона пациента.

В интерфейсе отображается:

+7 *** *** **45

Это исключает возможность прямого контакта врача с пациентом вне системы.

**Коммуникация через сервер-прокси**

Все сообщения отправляются через CRM.

Процесс:

1. врач пишет сообщение
2. сообщение отправляется на сервер
3. сервер отправляет сообщение пациенту через WhatsApp API

Врач не получает реальный номер пациента.

**Логи действий**

Система фиксирует ключевые действия пользователей:

- изменение статусов
- просмотр карточек
- отправка сообщений
- редактирование данных

Это позволяет:

- отслеживать подозрительную активность
- проводить аудит действий сотрудников

# **5. 2D карта зубов пациента**

**(клинический модуль MVP)**

В системе реализуется визуальная схема зубов пациента, позволяющая врачу удобно работать с состоянием зубов.

Отображается стандартная 2D стоматологическая карта зубов.

**Работа с картой зубов**

Каждый зуб является интерактивным элементом.

При нажатии на зуб открывается модальное окно, где врач может указать:

- состояние зуба
- диагноз
- комментарии
- рекомендации

**Действия с зубом**

В модальном окне доступны основные действия:

- начать лечение
- удалить зуб

Если зуб удаляется:

- он отображается на карте как удаленный
- это фиксируется в истории пациента

# **6. Автоматическое добавление препаратов и материалов**

При выборе процедуры система автоматически добавляет стандартный набор препаратов и материалов, используемых при данной операции.

Это ускоряет заполнение карточки процедуры.

**Автоматическое заполнение**

Для каждой операции задается шаблон препаратов.

Пример:

Удаление зуба:

- анестетик
- антисептик
- шовный материал
- обезболивающее

**Ручное добавление**

Врач может дополнительно указать:

- использованные материалы
- дополнительные препараты
- количество

Это обеспечивает гибкость учета медицинских процедур.
